# 1. Executive Summary

MatchMatrix — это личная production-платформа для анализа футбольных и хоккейных матчей через 30 гематрических моделей. Система предназначена для ручного и полуавтоматического анализа матчей, сохранения вычислений, сравнения моделей, ведения статистики, post-match проверки, retro-test анализа и постепенного самообучения через рекомендации Grok CLI.

Главная идея продукта — построить не просто калькулятор гематрии, а полноценную аналитическую систему:

- пользователь вводит или импортирует данные матча;
- система нормализует команды, составы, тренеров, капитанов, вратарей, судей, VAR и контекст турнира;
- 30 гематрических моделей рассчитывают значения;
- ансамбль моделей формирует прогноз по допустимым рынкам;
- после матча пользователь вводит счёт;
- система оценивает, какие модели дали точное, частичное или ошибочное направление;
- статистика сохраняется;
- Grok CLI анализирует ошибки и предлагает изменения весов;
- пользователь вручную подтверждает или отклоняет изменения.

На первом этапе система используется одним администратором, но архитектура должна сразу поддерживать будущую монетизацию, пользователей, подписки, тарифы, лимиты, роли, командный доступ и API-интеграции.

# 3. Product Goals

## 3.1 Главные цели

- Создать единую платформу для гематрического анализа футбола и хоккея.
- Исключить использование букмекерских коэффициентов.
- Исключить использование погоды.

- Поддерживать ручное подтверждение любых изменений весов.
- Подготовить архитектуру к будущей монетизации.

- 100% сохранение входных данных матча.
- 100% сохранение результатов всех активных моделей.

- Возможность сравнить модели по спорту, турниру, рынку и периоду.

Этот раздел фиксирует решения, подтверждённые владельцем проекта после discovery-сессии. Если другие разделы PRD звучат шире или старее, приоритет имеет этот раздел.

### 3.3.1 Product model

- MatchMatrix проектируется как личный продукт для одного владельца на старте.
- Архитектура сразу должна поддерживать будущих пользователей, роли, тарифы, лимиты, подписки, монетизацию и API-доступ.
- Первый пользовательский интерфейс — только закрытая локальная веб-панель администратора на `http://localhost:3100`.
- Публичная регистрация, личный кабинет пользователя, лендинг тарифов и платежи не входят в первые operational milestones, но учитываются в архитектуре.

### 3.3.2 Security and access

- Первый production-релиз должен иметь логин, пароль, защищённую session cookie и 2FA/TOTP.
- Административная панель не должна быть публичной без авторизации.
- Все рискованные действия требуют подтверждения: deploy, изменение production config, удаление данных, применение рекомендаций Grok, изменение весов и rollback.

### 3.3.3 Match data workflow

- Главный поток первой версии: ручной ввод матчей + интернет-поиск фактических данных.
- Интернет-поиск работает как полуавтоматический помощник: система предлагает найденные факты, пользователь подтверждает перед сохранением.
- Допустимы любые найденные страницы, если данные вручную подтверждаются владельцем.
- Импорт CSV/Excel/JSON не входит в первый operational milestone, но остаётся в архитектуре и поздних фазах production delivery.
- Post-match результат поддерживает два сценария: ручной ввод счёта и автоматический поиск счёта с подтверждением владельца.
- После подтверждения результата основной продукт хранит результат; технический audit может хранить служебный след, если это нужно для диагностики, но интерфейс не перегружается источниками.

### 3.3.4 Sports and analysis markets

- Фокус MVP/production v1 — результат игры и последующий разбор события.
- Для футбола остаются рынки `П1`, `X`, `П2`, `1X`, `12`, `X2`.
- Для хоккея остаются исходы основного времени `П1`, `X`, `П2` и итоговый победитель `П1`, `П2`.
- Тоталы, форы, обе забьют и букмекерские сценарии не добавляются в первую версию.

### 3.3.5 Two-language gematria layer

- Все ключевые данные и расчёты должны поддерживать два языковых слоя: English и Russian.
- Для российских и релевантных русскоязычных сущностей система хранит и считает обе версии раздельно.
- Языковые слои не смешиваются в одну строку: расчёты, trace, score, direction и confidence сохраняются отдельно.
- Каждые 10 матчей Grok анализирует, какой языковой слой даёт больше верных прогнозов и полезных расчётов.
- Интерфейс не должен перегружаться: dashboard показывает краткий вывод, а подробности доступны в полном разборе матча и статистике.
- Grok может предлагать изменить веса языковых слоёв или проверить гипотезу через retro-test; backend сам аналитические рекомендации не генерирует.

### 3.3.6 Gematria engine quality bar

- Все 30 гематрических моделей должны быть рабочими в production delivery, а не декоративно зарегистрированными.
- Для каждой модели сохраняется полный trace: исходные строки, язык, правила нормализации, символы, значения, промежуточные суммы, raw score, normalized score, direction, confidence, версия модели и вес на момент расчёта.
- Каждый запуск анализа сохраняет snapshot всех входных данных матча и настроек на момент расчёта.
- Если данные матча неполные, анализ разрешён, но система выставляет `data_quality`, показывает недостающие поля и снижает confidence.

### 3.3.7 Grok CLI operating model

- Grok CLI работает через авторизацию владельца/OAuth и подписку Grok Super.
- Grok должен работать полноценно как AI-аналитик, но через контролируемый `grok-worker`, очередь задач, backend API и audit log.
- В MVP обязательны ручной запуск Grok CLI и автоматический запуск после post-match.
- Расписание, автозапуск после retro-test и автозапуск после импорта проектируются, но не являются первыми критическими сценариями.
- Grok может читать данные матча, расчёты, trace, статистику, историю и заметки, если это нужно для анализа.
- Grok может создавать черновики рекомендаций, отчётов, комментариев, новых моделей и изменений весов.
- Grok не применяет изменения сам: каждое изменение веса, модели, промпта или настройки требует ручного подтверждения владельца.

### 3.3.8 Interface decisions

- Dashboard — рабочая очередь из максимум 10 активных матчей, а не обычный список всех матчей.
- Приоритет очереди: неполные данные, готовые к pre-match, ожидающие результата, ожидающие post-match разбора, свежие завершённые с разбором.
- Полный список матчей находится в разделе `Матчи`.
- Визуальный стиль — премиальный, тёмный, спокойный, ближе к Apple/Tesla, с аккуратным вау-эффектом.
- Аналитические таблицы и графики должны быть функциональными, но не превращать интерфейс в перегруженный терминал.
- Полный разбор матча открывается как полноэкранное модальное окно внутри dashboard.
- Внутри полного разбора должны быть все аналитические данные по матчу и гематрии: итог, языковые слои, 30 моделей, trace, веса, Grok-разбор, post-match сравнение и предложения улучшений.
- Заметки по матчу и заметки по модели входят в первую production-версию и доступны прямо в полном разборе.
- Экспорт Markdown/PDF/CSV не входит в первый operational milestone; основной отчёт — экран внутри системы.

### 3.3.9 Backup and local runtime

- В первом operational milestone backup запускается ручной кнопкой `Создать backup`.
- Автоматический daily backup проектируется, но не является обязательным первым сценарием.
- Проект должен работать полностью локально с ПК владельца.
- VPS deploy, внешний Nginx, публичный домен, DNS и production TLS не требуются.
- MatchMatrix запускается как локальный Docker Compose stack с project name `matchmatrix`.
- Локальный frontend доступен по `http://localhost:3100`.
- Локальный backend API доступен по `http://localhost:8100`.
- PostgreSQL и Redis запускаются локально в Docker volumes.
- Перед запуском обязателен local preflight-check: Docker доступен, localhost-порты свободны, `.env` настроен, volumes существуют, Grok CLI доступен.
- Если в будущем владелец снова захочет VPS, это будет отдельное решение и отдельная фаза, не часть текущего scope.

# 9. Core Modes

## 9.1 Pre-match

Pre-match — режим анализа до начала матча.

Шаги:

1. Пользователь создаёт матч.
2. Заполняет обязательные поля.
3. При неизвестном составе выбирает статус “предварительный состав”.
4. Система нормализует данные.
5. Гематрический движок считает 30 моделей.
6. Weight Engine применяет веса.
7. Ensemble Engine формирует вывод.
8. Grok CLI объясняет прогноз.
9. Результат сохраняется.

## 9.2 Post-match

Post-match — режим после завершения матча.

Шаги:

1. Пользователь вводит счёт.
2. Система определяет фактический исход.
3. Система сравнивает прогнозы моделей с фактом.
4. Каждая модель получает статус:
   - exact;
   - partial;
   - wrong;
   - no_signal.
5. Обновляется статистика.
6. Запускается Grok Error Analysis.
7. Формируются рекомендации.
8. Пользователь подтверждает или отклоняет изменения.

## 9.3 Retro-test

Retro-test — режим проверки моделей на исторических матчах.

Шаги:

1. Пользователь загружает исторические матчи.
2. Система считает модели так, как будто результат неизвестен.
3. Затем сравнивает с фактическим счётом.
4. Формирует рейтинг моделей.
5. Показывает статистику по периодам, турнирам и видам спорта.

# 20. 30 Gematria Models


## 20.1 English Ordinal

**Формула:** `A=1 ... Z=26`

**Описание:**  
Базовая англоязычная порядковая система. Хороша как контрольная модель и базовый слой для сравнения.

**Алгоритм:**

1. Получить нормализованный текст.
2. Удалить неподдерживаемые символы или преобразовать их по таблице.
3. Рассчитать значение каждого символа.
4. Сложить значения.
5. Сохранить посимвольный trace.
6. Рассчитать raw score.
7. Рассчитать normalized score.
8. Сравнить home и away значения.
9. Сформировать direction.
10. Передать результат в ансамбль.

**Вес по умолчанию:** `1.0`

**Сильные стороны:**

- Даёт отдельный независимый числовой сигнал.
- Может использоваться в ансамбле.
- Сохраняет trace для аудита.

**Слабые стороны:**

- Не должна использоваться как единственный источник решения.
- Может переобучаться на малом числе матчей.
- Требует статистической проверки после каждого результата.

**Хранение результата:**

- `model_id`
- `match_id`
- `input_payload`
- `normalized_payload`
- `calculation_trace`
- `raw_score`
- `normalized_score`
- `direction`
- `confidence`

**Статистика:**

- точные попадания;
- частичные попадания;
- ошибки;
- средний confidence;
- эффективность по спорту;

- Каждая модель имеет вес.
- Вес влияет на ансамбль.
- Вес может быть изменён только пользователем.
- Grok CLI может только рекомендовать изменение.
- История веса должна сохраняться.

## 21.2 Формула ансамбля

Упрощённая формула:

```text
ensemble_score = SUM(model_signal * model_weight * confidence) / SUM(active_model_weights)
```

## 21.3 Рекомендации веса

Grok CLI может предложить:

- повысить вес;
- снизить вес;
- отключить модель;
- создать комбинированную модель;
- протестировать модель на retro-test.

# 33. Phased Production Delivery Roadmap

Проект делается сразу как полноценная production-система, но поставляется фазами. Каждая фаза должна иметь контрольную точку, проверку качества и понятный критерий готовности. Нельзя переходить к следующей фазе, если текущая не проходит build/test/security/preflight проверки.

## Phase 0 — Repository and Safety Baseline

Цель: создать безопасный фундамент проекта для локального запуска без VPS.

Включает:

- монорепозиторий или согласованную структуру `frontend`, `backend`, `worker`, `infra`, `docs`;
- локальный `AGENTS.md` из глобальных инструкций;
- README с архитектурой и командами;
- `.env.example` без секретов;
- Conventional Commits;
- secret-scan перед commit;
- базовый Docker Compose skeleton без конфликта с локальными портами.

Контрольная точка:

- проект открывается локально;
- `docker compose config` проходит;
- нет секретов в репозитории;
- структура готова к backend/frontend разработке.

## Phase 1 — Backend, Database and Auth Foundation

Цель: заложить backend foundation для личной admin-панели.

Включает:

- FastAPI backend;
- PostgreSQL schema и миграции;
- Redis подключение;
- модели users/roles/sessions/2FA/audit_logs/system_settings;
- логин, пароль, session cookie;
- TOTP 2FA;
- базовый audit log;
- healthcheck endpoints;
- backend tests.

Контрольная точка:

- администратор может войти только после 2FA;
- backend не стартует без обязательных env;
- PostgreSQL и Redis работают через Compose;
- auth endpoints покрыты тестами.

## Phase 2 — Match Management and Fact Assistant

Цель: дать владельцу рабочий поток создания матча.

Включает:

- CRUD матчей;
- команды, турниры, арены, судьи, VAR, составы, тренеры, капитаны, вратари;

- web-fact assistant для поиска фактов;
- подтверждение найденных фактов перед сохранением;
- post-match result manual entry;
- automatic result lookup with confirmation.

Контрольная точка:

- владелец вручную создаёт матч;
- система помогает найти факты;
- неполные данные не блокируют анализ, но снижают confidence;
- результат матча можно ввести или подтвердить после поиска.

## Phase 3 — Gematria Engine v1: 30 Working Models

Цель: реализовать полный рабочий гематрический движок.

Включает:

- 30 рабочих моделей;

- snapshot входных данных;
- model versioning;
- weights;

Контрольная точка:

- локальный запуск можно повторить без ручного хаоса.

Цель: подготовить продукт к будущему открытию доступа без включения публичной монетизации в первую эксплуатацию.
